# FlipkartEcommerce — DevOps Delivery Roadmap

*Companion to the FlipkartEcommerce DevOps Architecture & Delivery Plan.*

This document lays out the **hands-on build order** for the platform. Each stage
adds exactly one new capability on top of the previous one, so by the end you go
from an empty cloud project to a fully automated, self-managing delivery system.
The idea is to learn (and demo) one concept at a time instead of wiring
everything up at once.

---

## The Roadmap (basic to advance)

| # | Stage | Full toolchain *(new tool in bold)* | You end up with |
|---|-------|-------------------------------------|-----------------|
| 1 | Infrastructure as Code | **GitHub** + **Terraform** + **GCP** | Networks, VMs, GKE, Cloud SQL — all reproducible |
| 2 | Host configuration | GitHub + Terraform + GCP + **Ansible** | Jenkins, SonarQube, Docker installed and wired up |
| 3 | Containerized CI/CD | GitHub + Terraform + GCP + Ansible + **Maven** + **npm** + **SonarQube** + **Jenkins** + **Docker** + **Docker Hub** | A `fec-*` service running as a Docker container |
| 4 | Kubernetes delivery | GitHub + Terraform + GCP + Ansible + Maven + npm + SonarQube + Jenkins + Docker + Docker Hub + **Kubernetes (GKE)** | The same service running on GKE, scaled and self-healing |
| 5 | Reusable pipelines | GitHub + Terraform + GCP + Ansible + Maven + npm + SonarQube + Jenkins + Docker + Kubernetes (GKE) + **Jenkins Shared Libraries** + **JFrog Artifactory** | One shared pipeline every repo reuses |
| 6 | Templated deploys | GitHub + Terraform + GCP + Ansible + Maven + npm + SonarQube + Jenkins + Docker + Kubernetes (GKE) + Jenkins Shared Libraries + JFrog Artifactory + **Helm** | Each service packaged as a Helm chart |
| 7 | Cluster automation | GitHub + Terraform + GCP + Ansible + Maven + npm + SonarQube + Jenkins + Docker + Kubernetes (GKE) + Jenkins Shared Libraries + JFrog Artifactory + Helm + **Kubernetes Admin Tasks** | Routine cluster admin done by pipelines |

> **Registry note:** stages 3–4 use a simple registry (**Docker Hub**) while
> you're learning; from stage 5 onward it's replaced by **JFrog Artifactory** —
> the same registry used in the architecture plan — once pipelines are shared and
> production-like.

## Stage 1 — Infrastructure as Code (Terraform)

Nothing gets built by hand. Before any application work begins, the entire GCP
environment is described in Terraform and stored in the infrastructure GitHub
repo. Terraform stands up the networking (VPC, subnets, firewall rules), the
compute hosts that will run Jenkins / SonarQube / Docker, the **GKE** cluster for
the `fec-*` workloads, and a **Cloud SQL (MySQL)** instance for the backends.
Because it's all code, the whole environment can be torn down and recreated
identically at any time.

## Stage 2 — Host Configuration (Ansible)

Terraform gives you bare machines; Ansible turns them into working tools. Once the
VMs exist, Ansible playbooks (also in GitHub) install and configure the **Jenkins
master and agents**, the **SonarQube** server, **Docker**, and any supporting
packages. Splitting this from stage 1 keeps a clean line between *provisioning*
(Terraform) and *configuration* (Ansible) — and the database is deliberately left
out of Ansible's reach.

## Stage 3 — Containerized CI/CD

Now the application enters the picture. A Jenkins pipeline takes a `fec-*` service
through build (**Maven** for the Java services, **npm** for the React apps), a
**SonarQube** quality scan, and then packages the result as a **Docker** image.
The image is pushed to a container registry and run as a container. This is the
first end-to-end "commit to running service" loop.

## Stage 4 — Kubernetes Delivery

Running loose containers doesn't scale, so this stage points the pipeline at
**GKE**. The build and scan steps stay the same, but instead of `docker run`, the
image is deployed to the Kubernetes cluster — which handles scaling, restarts, and
high availability for you. The same `fec-*` service now lives as a managed
workload.

## Stage 5 — Reusable Pipelines (Jenkins Shared Libraries)

By now several services have nearly identical pipelines, which is a maintenance
trap. A **Jenkins shared library** extracts the common build/scan/package/deploy
logic into one versioned place that every `fec-*` repo imports. At the same point,
image storage moves to **JFrog Artifactory** for proper artifact management. Change
the pipeline once, and every service benefits.

## Stage 6 — Templated Deployments (Helm)

Raw Kubernetes manifests are hard to reuse across environments. **Helm** packages
each service's manifests into a chart with values you can override, so the same
chart deploys to dev, staging, or prod — and gives you versioned releases plus
one-command rollbacks when something goes wrong.

## Stage 7 — Automating Cluster Operations

The final stage turns repetitive day-2 cluster chores into pipelines. Using
Jenkins and the shared library, routine **Kubernetes administrative tasks** —
scaling workloads, cleaning up resources, applying network policies — are
triggered as jobs instead of typed by hand, keeping the cluster consistent and
freeing the team from manual `kubectl`.

---

## The Big Picture

Read top to bottom, the seven stages form one continuous story: **provision with
Terraform, configure with Ansible, build and containerize with Jenkins + Docker,
run on GKE, standardize with shared libraries, release with Helm, and finally let
pipelines run the cluster itself.** Each stage is small enough to learn on its own,
but stacked together they deliver a fully automated, scalable, and reusable
delivery platform for FlipkartEcommerce.
